Armv7m Startup

By Martin Ribelotta | September 5, 2020

One of the most successful families microcontrollers has been the cortex-m in its various versions (m0, m0+, m3, m4, m7, m33, etc), and one of the reason of these success is the simplest boot process. In this article, we travel around the magic of code writing and see the different implementations across some vendors and propose a vendor-neutral startup schema with many advantages for your projects.

In this series of articles I try to explain the tricks around code compilation, linker scripts and build systems, proposing, additionally, a vendor neutral startup system for your cortex-m project.

In the way of universal cortex-m bootstrap

The boot process for the cortex-m series is straight forward:

  • On the reset, the processor take the value of the special memory-mapped register VTOR and use this as a pointer to “interrupt table”.
  • The first 32 bits of this table is loaded in the register SP for point the processor stack
  • The second 32 bits is loaded in register PC and point to start of the user code.
  • The processor start executing the instruction pointed by the current PC (the value previously loaded from VTOR+4 table)

This schema, is really simple but powerful, this enables the user code to be writes entirely in C following the binary interface of ARM processors (AAPCS standard) followed by virtually any compiler for this architecture.

In contrast, the bast majority of firmware uses assembler routine for the main startup sequence for simplicity and control. in fact, this speak about the trust in the compiler that the chip vendor have. The main reason for this is to provide a consistent boot code across different compilation parameters and optimizations, because some compiler bugs in the past break the startup routine in weird manners. At this time, the serious compilers like gcc or clang (and the arm tunned version: armcc6) have a regression test battery as part of continuos integration process and the weird bugs in common code is really rare.

Writing simple startup…

First of all, you need a vector with pointers to exception handlers, reset handler and the initial stack pointer:

extern unsigned int _stack; /* Defined by the linker */

/* Function prototype, defined after */
void reset_handler(void);
void hang_isr(void);
void pass_isr(void);

void *vector_core[] __attribute__((section(".vector_core"))) = {
    (void*) &_stack,
    (void*) reset_handler,
    (void*) hang_isr, /* NMI */
    (void*) hang_isr, /* Hard Fault */
    (void*) hang_isr, /* Memory Manager Fault */
    (void*) hang_isr, /* Bus Fault */
    (void*) hang_isr, /* Usage Fault */
    (void*) 0UL,
    (void*) 0UL,
    (void*) 0UL,
    (void*) 0UL,
    (void*) 0UL,
    (void*) 0UL,
    (void*) 0UL,
    (void*) pass_isr, /* SVC Call */
    (void*) 0UL,
    (void*) 0UL,
    (void*) pass_isr, /* Pend Supervisor Call */
    (void*) pass_isr, /* SysTick */
};

This vector is marked to belong into a code section called “.vector_core”, the linker (via linking scripts) will ensure to place it in the reset vector of the chip, either the one identified by the manufacturer or the one specified by ARM default.

In fact, the vector table is more big than this common entries but the rest of vectors is vendor specific and need to handle more robust. In all implementations, the vector table is a big array with the first 16 entries common of all cortex-m parts (ARMv7M specifications).

The hang action take on severe condition ISR is to halt the processor and is very simple, an infinite loop.

void hang_isr(void)
{
    while (1) {} /* Or if you love code obfuscation try this: for(;;); */
}

The do-nothing handler is only an empty function

void pass_isr(void)
{
}

Ideally you be provide an individual function for all of exception type and store this in your own file (one for hard fault, one for mem fault, one for systick, etc) but, for the example, this is a valid approach.

Finally, the reset handler is the start point of your program:

void reset_handler(void)
{
    /* Your program here */
    while (1) {

    }
}

Ideally, at this point, you can execute any C code, but some extra work is required due to provide a perfect standard-fits C environment.

In the reset_handler function some thinks is not compliance:

  • The C language specify that all uninitialized data must be zeroed by the startup code or OS before main function. This is called BSS section.
  • Uninitialized data has not been copied from ROM to RAM. These operation must be done before main. This is called DATA section.
  • Some C constructions emit initialization code that requires manually executed before main function. These code must need to called in this point. This is essential for C++ compatibility, virtually all constructors of static class is called here.

In order to complete the pre-main code, the compiler need some help of the linker script to find these memory address:

  • Start of initialized data in RAM: _data
  • End of initialized data in RAM: _edata
  • Start of the copy of initialized data in ROM: _data_loadaddr
  • Start of BSS address
  • End of BSS address
  • Start of array with initialization function pointers __init_array_start;
  • End of array with initialization function pointers __init_array_end;

These symbols must be provided by the linker script and, for the C compiler, this is viewed as an “extern defined unsigned integer”. The real type of these symbols may be anything, but unsigned int is usefully to work with the natural type of data (in cortex-m, int match the register size)

extern unsigned int _data_loadaddr;
extern unsigned int _data;
extern unsigned int _edata;
extern unsigned int _bss;
extern unsigned int _ebss;
extern unsigned int __init_array_start;
extern unsigned int __init_array_end;

With these data defined in the linker script you can process to all task for prepare to main:

/* Define a convenient type for cast integer ptr to function ptr */
typedef void (*func_t)(void);

/* Prototype of main function */
extern int main(void);

void reset_handler(void)
{
    unsigned int *src;
    unsigned int *dst;

    /* Copy data from rom to ram */
    src = &_data_loadaddr;
    dst = &_data;
    while (dst < &_edata) {
        *dst++ = *src++;
    }

    /* Set to 0 all uninitialized data */
    dst = &_bss;
    while (dst < &_ebss) {
        *dst++ = 0;
    }

    /* Call all startup functions */
    src = &__init_array_start;
    while (src < &__init_array_end) {
        ((func_t)src++)();
    }

    /* Finally, call main */
    main();

    while (1) {} /* And hang if main return */
}

This work fine in all optimizations and across al cortex-m implementations, but is perfectible.

Play the game… of compiler optimization

The first think when you saw “optimization” is go for all, enable all heavy optimization flags, all code tricks and see the result.

In gcc this is says as -O3 and the result is not disappoint:

But this is very different compared with the hand-made code:

First of all you see a sentence like push {r3, r4, r5, lr}. This is clobber register save because the compiler try to create a normal function callable by any code, but this function is special because is the first function called by the hardware… This disappoint is fixed adding a naked attribute to the function. From the gcc manual:

Use this attribute on the ARM, AVR, MCORE, RX and SPU ports to indicate that the specified function does not need prologue/epilogue sequences generated by the compiler.

Additionally, the same part of gcc documentation says that is not safe include any statement distinct of asm but for these purpose you can ignore this warning and go for all ¯\_(ツ)_/¯. In clang, the compiler end with an error if your naked function try to use anything apart of asm statement.

Another point is the requirement of memcpy and memset functions. Yeeees, these ar standard functions and may be provided by virtually any compiler or c library in the world, but in a freestanding environment (without OS or without runtime) this may be undesirable behavior…

The rationale of this compiler decision is that:

  • mem* functions is hand made assembler optimized and potentially more efficient in large data move/set
  • Maybe perform unaligned set/move faster than C hand-made function.
  • Both functions is standard even if you infer freestanding environment

If you are ok with this points, the -O3 trick is really usefully, but in certain cases is not desirable:

  • You may not need to link with libc or provide mem* implementation.
  • Your mem* implementations really sucks.
  • Your data and bss is tiny and the overhead in code and execution type for memcpy/memset is great than optimal.
  • You are a friki of optimization and write articles with a lot of assembler on your blog in the weekends.

Exploring custom optimizations

All optimizations are a compromise, this is not a exception… if you find a binary code most like a hand-made code you need some points of anchor:

  • The data and bss sections start and end word aligned.
  • The most efficient way to move data using the processor is in word aligned manner. This implies the load->store pair of instruction take the best advantage of data bus bandwidth.
  • The processor not have complex pipeline with advance superscalar architecture or out-of-order and speculative execution, and consequently all optimizations about the instruction pipe exploit is futile or less efficiently than little code.
  • Small loops fit in execution buffer or similar structure and perform execution more efficient than linear execution in an speculative pipe

All above assumptions is nearly true in the most common ARMv7M implementations like cortex-m0 to cortex-m7 and consequently, the best optimizations is not necessarily the same as for larger processors.

In this scenario, a less aggressive optimization perform more efficiently than -O3. By example, is desirable that the machine code follows as closely as possible, the idea expressed in the source code, and the best way to do this in modern version of gcc and clang is the -Og optimization.

You can check the code result of our startup code using -Og and naked attribute here:

Reload constants in every loop

The bit of C code:

  src = &_data_loadaddr;
  dst = &_data;
  while (dst < &_edata) {
    *dst++ = *src++;
  }

Is translated into

        ldr     r3, .L9    ; dst = &_data;
        ldr     r2, .L9+4  ; src = &_data_loadaddr;
.L3:
        ldr     r1, .L9+8  ; &_edata
        cmp     r3, r1     ; dst < $r1? <- this load is performed in every loop
        bcs     .L7
        ldr     r1, [r2], #4
        str     r1, [r3], #4
        b       .L3
.L7:

By default, the -Og optimization not enables loop invariant extraction.

For know more: Loop invariant extraction is a technique to extract all non-changed data outsize of a loop. This perform better loop speed at cost of non perfect match of the final C code.

See: https://en.wikipedia.org/wiki/Loop-invariant_code_motion

The not inclusion of loop invariant extraction from -Og is a philosophical debate but for this purpose the loop invariant optimization is very desirable in any case because maintain the behavior of source code and improve loop performance.

For enable this, you need to use -fmove-loop-invariants flag added to -Og, and the complete result can see here

Concretely, the chunk of code is compiled now like this:

        ldr     r3, .L9   ; dst = &_data;
        ldr     r2, .L9+4 ; src = &_data_loadaddr;
        ldr     r1, .L9+8 ; &_edata
.L3:
        cmp     r3, r1    ; dst < &_edata?
        bcs     .L7
        ldr     r0, [r2], #4 ; *src++;
        str     r0, [r3], #4 ; *dst++ = ...
        b       .L3
.L7:

Very near to hand-write code. Same analysis can be made for zero fill bss code.

Minor fixes with reorder code

If you examine the last piece of code before main call you can see that the compiler can do better:

    /* Call all startup functions */
    src = &__init_array_start;
    while (src < &__init_array_end) {
        ((func_t)src++)();
    }

Is translate as:

.L8:
        ldr     r3, .L9+20  ; $r3 is src <= &__init_array_start;
        ldr     r4, .L9+24  ; $r4 <= &__init_array_end
.L4:
        cmp     r3, r4      ; (src < &__init_array_end)?
        bcs     .L5
        ; Hey gcc! you can do this better...
        adds    r5, r3, #4  ; $r5 <= src++
        blx     r3          ; jump to [$r5]
        mov     r3, r5      ; Restore $r3 with temporary $r5
        b       .L4

The main reason to this behavior is the rule *as-is, in other word, gcc try to optimize the code maintain exactly the C code structure. If you analyze the centene ((func_t)ptr++)() you can see that the code follow perfectly the code:

  1. Take the value of pointer ptr
  2. Increment ptr
  3. Call the function pointed by the value taken in the step 1

The behavior is, first increment the pointer, and next call the function, other behavior is not the correct way to generate the code (at least in -Og)

The simplest way to improve generated code is reorder the source in less fancy way:

    for (unsigned int *src = &__init_array_start; src < &__init_array_end; src++) {
        ((func_t)src)();
    }

And the generated code is:

        ldr     r4, .L4    ; $r4 <= &__init_array_start
        ldr     r5, .L4+4  ; $r5 <= &__init_array_end
.L3:                       ; for(...) {
        cmp     r4, r5     ;   src < &__init_array_end?
        bcs     .L1
        blx     r4         ;     ((func_t)src)();
        adds    r4, r4, #4 ;   src++
        b       .L3        ; }
.L1:

As rule of thumb, for is more optimizable than the equivalent forms in code because the compiler contains many techniques for detect cycles and perform transformations on that… Help the compiler, write in non-fancy/non-flamboyant manner.

The hex of scripts: Linker script

As you have read, the startup code need to know some information about the memory layout, specifically, the start and end of data and bss sections, apart of the start of data initialized data in ROM.

This information is provided by the linker when it combine all object files to produce the final binary. But the memory layout is very dependent of the platform… not all devices have the flash and RAM in the same space.

The elegant way chosed by gcc and gnu tools to handle this is the linker scripts, a one or more text files describing the particularities of memory layout, section placement and other information (like symbol need to defined) required to produce the final executable.

A fast review

The gnu toolchain build process is like this:

  • All *.c files is build individually to an object file *.o. These files is a collection of sections containing binary data of diferet types:
    • The code of every C file generate .text section
    • The read only data generate .rodata section
    • The non zero initialized data generate .data section
    • The zero initialized data generate .bss section
    • The non initialzed data generate COMMON section and C standard mandate to initialize in 0 like .bss section. By this reason, some compilers only generate .bss section joining both sections.
  • The linker join individual object sections and put joined sections in specific address depending of the linker script layout.
  • If you not provide linker script, the toolchain provide one by default. Normally, this script is not suitable to use in bare metal devices.

The linker script consist in commands to describe the placement of the diferent sections in memory area.

In the next code, you can see a complete functional linker script compatible with the startup exposed in last sections:

/* Start of section description layout */
SECTIONS
{
  . = 0x08000000; /* The dot is current address counter. */

  /* If you not specify any, start at 0 */
  .text : {
    /* KEEP ensure that symbols is not discarded if the linker
       is insturde to remove unised symbols. */
    KEEP(*(.vector_core*)) /* Place .vector_core at start of ROM */
    *(.text)   /* Place all symbols in ".text" section */
    *(.text.*)  /* Place all symbols start with section ".text.<any>" */
  }

  /* In many platforms, the linker script put the rodata in text if
     the corresponding section not exist */
  .rodata : {
    . = ALIGN(4); /* Ensure word aligment of sections */
    *(.rodata)
    *(.rodata*)
    . = ALIGN(4); /* Ensure word aligment of sections */
  }

  .init_array : {
    . = ALIGN(4); /* Ensure word aligment of sections */
    __init_array_start = .;
    /* SORT function sort the symbols listed in alphabetical order.
       This is required to meet with the initialization order.
       All symbols is in the form of .init_array.0000_nameofsymbol
       when 0000 is replaced by a priority number that can be specified
       at compile time. */
    KEEP(*(SORT(.init_array.*)))
    KEEP(*(.init_array))
    . = ALIGN(4); /* Ensure word aligment of sections */
    __init_array_end = .;
  }

  _etext = .; /* You can define symbols with any expression using dot */

  . = 0x20000000;
  /* AT(expr) is used to specify load address */
  .data : AT(_etext) {
    _data = .;
    *(.data)
    *(.data*)
    . = ALIGN(4);
    _edata = .;
  }

  /* The function LOADADDR return the load address of specific section */
  _data_loadaddr = LOADADDR(.data);

  .bss : {
    . = ALIGN(4); /* Ensure word aligment of sections */
    _bss = .;
    *(.bss)
    *(.bss*)
    *(COMMON)
    . = ALIGN(4);
    _ebss = .;
  }

  /* You use prefix like K for multiply by 1024 or M for multiply for 1048576 */
  _stack = 0x20000000 + 20K; /* Put the stack at end of ram (20K in this exaple) */
}

The only non-evident magic in this script is the AT(_etext) expression. This instrude the lenker to put the section in specific load address but for the program, this is mapped to current dot address. This is specifically designed to place initialize data in ROM and copy this to RAM at startup.

this linker script supposes that the rom memory start of 0x08000000, the ram address start at 0x20000000 and end 8K later (the case for the common bluepill processor stm32f103cb)

You can improve these script adding more user friendly constants:

ROM_BASE = 0x08000000;
RAM_BASE = 0x20000000;
RAM_SIZE = 20K;

/* Start of section description layout */
SECTIONS
{
  . = ROM_BASE;
  
  ...

  . = RAM_BASE;
  /* AT(expr) is used to specify load address */
  .data : AT(_etext) {
    _data = .;

  ...

  _stack = RAM_BASE + RAM_SIZE;
}

Or move this to another platform depend file:

INCLUDE(memory.ld)

/* Start of section description layout */
SECTIONS
{
  . = ROM_BASE;
  ...

/* Platform dependent memory.ld */
ROM_BASE = 0x08000000;
RAM_BASE = 0x20000000;
RAM_SIZE = 20K;

But the realy nice form to put diferent sections in memory areas is via MEMORY command. You can replace the content of memory.ld with:

MEMORY {
  rom rx : ORIGIN 0x08000000, LENGTH = 64K
  ram rwx : ORIGIN 0x20000000, LENGTH = 20K
}

And the new linker script need some changes to use the memory regions properly:

INCLUDE memory.ld

SECTIONS
{
  .text : {
    KEEP(*(.vector_core*))
    *(.text)
    *(.text.*)
  } > rom /* fill rom memory area with .text section */

  .rodata : {
    . = ALIGN(4);
    *(.rodata)
    *(.rodata*)
    . = ALIGN(4);
  } > rom /* the second >rom append to previos block the current */

  .init_array : {
    . = ALIGN(4);
    __init_array_start = .;
    KEEP(*(SORT(.init_array.*)))
    KEEP(*(.init_array))
    . = ALIGN(4);
    __init_array_end = .;
  } > rom

  .data : {
    _data = .;
    *(.data)
    *(.data*)
    . = ALIGN(4);
    _edata = .;
  } > ram AT>rom /* AT>section is to append in load memory address mode */

  _data_loadaddr = LOADADDR(.data);

  .bss : {
    . = ALIGN(4);
    _bss = .;
    *(.bss)
    *(.bss*)
    *(COMMON)
    . = ALIGN(4);
    _ebss = .;
  } > ram

  /* ORIGIN(region) return the address of memory region, LENGTH return the length */
  _stack = ORIGIN(ram) + LENGTH(ram);
}

This trick have some advantages:

  • The linker can check automatically when memory section is exausted and end with error
  • You no need to care for maintain the dot pointer in the correct value
  • The linker script explain autoamtically where is your code
  • New versions of ld can display in user friendly manner compsumintion for all memory areas
